## 3. 生成 HTTP 请求消息

### 3.1. 浏览器解析 URL

当用户在浏览器中输入一个 URL 并按下回车时，浏览器首先要对 URL 进行解析，确定请求的目标服务器和资源路径。URL 的格式通常为「协议://域名/路径」，例如 `https://www.example.com/index.html`。浏览器从中提取出协议类型（如 HTTP 或 HTTPS）、服务器域名（如 www.example.com）以及资源路径（如 /index.html）。如果 URL 中没有指定路径，浏览器会自动补充一个默认文件名，通常是 index.html。

### 3.2. HTTP 请求的基本流程

浏览器生成 HTTP 请求消息的基本流程可以概括为四个步骤。第一步是解析 URL，提取服务器域名和资源路径。第二步是生成 HTTP 请求消息，包括请求行（方法、路径、版本）、请求头和消息体。第三步是向 DNS 服务器查询 Web 服务器的 IP 地址，因为网络通信需要的是 IP 地址而非域名。第四步是委托操作系统的协议栈，通过 TCP/IP 协议将消息发送到服务器。

### 3.3. 生成请求消息

HTTP 请求消息由请求行、请求头和空行、消息体三部分组成。请求行包含请求方法（如 GET、POST）、资源路径和 HTTP 版本号，各部分之间用空格分隔。请求头包含各种附加信息，如 Host（目标主机名）、Content-Type（消息体格式）等，每行一个字段，格式为「字段名: 值」。请求头和消息体之间用一个空行分隔。GET 请求通常没有消息体，POST 请求的消息体中包含要发送给服务器的数据。

## 4. 向 DNS 服务器查询 IP 地址

### 4.1. IP 地址的基本知识

IP 地址是网络中用于标识设备的数字编号。IPv4 地址由 4 个字节（32 位）组成，通常用点分十进制表示，如 192.168.1.1。每个字节的取值范围是 0 到 255。在实际通信中，IP 地址还需要配合子网掩码来判断目标设备是在同一子网内还是在外部网络。

### 4.2. 域名与 IP 地址的关系

域名是人类可读的网站地址（如 www.example.com），而 IP 地址是网络设备实际使用的数字编号。两者之间通过 DNS（Domain Name System）进行映射。浏览器在发送 HTTP 请求之前，必须先通过 DNS 查询将域名转换为对应的 IP 地址，否则无法建立网络连接。

### 4.3. Socket 库与解析器

DNS 查询的实际工作由操作系统中的 Socket 库完成。Socket 库中包含一个名为解析器（resolver）的组件，专门负责 DNS 查询。浏览器调用解析器并传入域名，解析器向 DNS 服务器发送查询请求，收到响应后将 IP 地址返回给浏览器。解析器本身也是一段程序代码，通过操作系统的协议栈发送和接收 DNS 查询用的 UDP 数据报。

## 5. 全世界 DNS 服务器的大接力

### 5.1. DNS 服务器的基本工作

DNS 服务器中存储着域名与 IP 地址的映射记录。当收到查询请求时，DNS 服务器会在自己的数据库中查找是否有匹配的记录。如果找到了，就直接返回对应的 IP 地址；如果没有找到，则将请求转发给其他 DNS 服务器继续查询。一台 DNS 服务器无法查询到所有域名的信息，因此需要多台 DNS 服务器接力配合完成查询。

### 5.2. 域名的层次结构

互联网上的域名采用层次化的组织方式。以 www.example.com 为例，从右向左依次是顶级域（com）、二级域（example）和三级域（www）。每个域都可以进一步划分为子域，形成树状结构。这种层次化的设计使得 DNS 系统能够分布式地管理海量的域名信息，不同层次的域名由不同的 DNS 服务器负责管理。

### 5.3. 寻找对应的 DNS 服务器

查询域名时，客户端首先联系根 DNS 服务器。根 DNS 服务器知道所有顶级域（如 com、org、cn）的 DNS 服务器地址，它会告诉客户端去查询哪个顶级域 DNS 服务器。顶级域 DNS 服务器再指引客户端去查询负责该二级域的 DNS 服务器。这样逐级向下，最终找到存储目标域名记录的权威 DNS 服务器，获得 IP 地址。

### 5.4. 通过缓存加快 DNS 服务器的响应

DNS 查询每次都要经过多级服务器的接力会很慢，因此 DNS 服务器引入了缓存机制。当 DNS 服务器成功查询到某个域名的 IP 地址后，会将结果保存在缓存中一段时间。下次收到相同域名的查询时，直接从缓存中返回结果，避免了重复的逐级查询。缓存的保存时间由 DNS 记录中的 TTL（Time To Live）字段控制，过期后缓存会被清除，下次查询时需要重新向权威服务器获取最新记录。

## 6. 委托协议栈发送消息

### 6.1. 协议栈的结构

浏览器拿到 IP 地址后，接下来要委托操作系统的协议栈来完成实际的网络通信。协议栈分为多个层次：上层是 TCP 和 UDP 等运输层协议，负责数据的可靠传输或快速传输；下层是 IP 协议，负责将数据包发送到目标地址；再往下是驱动程序层，负责控制网卡等硬件设备。这种分层结构使得每一层只需关注自己的职责。

### 6.2. 创建套接字

通信的第一步是创建套接字（socket）。套接字是协议栈内部的一种数据结构，用于记录通信的状态和参数。浏览器调用 Socket 库中的 socket 函数，协议栈会分配相应的资源并返回一个套接字描述符。这个描述符在后续的操作中用来标识这个套接字。创建套接字时还没有建立连接，套接字中只记录了通信的基本信息。

### 6.3. 连接服务器

创建套接字后，客户端调用 connect 函数与服务器建立 TCP 连接。这个过程就是著名的三次握手：客户端先发送一个 SYN 包表示希望建立连接；服务器回复 SYN+ACK 包表示同意连接；客户端再发送一个 ACK 包确认连接建立。连接建立后，套接字中会记录双方的 IP 地址和端口号等信息，为后续的数据传输做好准备。

### 6.4. 收发数据

连接建立后，客户端调用 write 函数将 HTTP 请求消息交给协议栈。协议栈将数据分割成合适大小的块，添加 TCP 首部和 IP 首部，通过网卡发送出去。服务器收到数据后，协议栈回复 ACK 确认。客户端调用 read 函数接收服务器的 HTTP 响应，协议栈从接收到的数据包中提取出消息内容并返回给浏览器。整个收发过程中，协议栈会根据 TCP 协议的规则进行流量控制和拥塞控制。

### 6.5. 断开连接并删除套接字

数据收发完成后，需要断开 TCP 连接。断开过程通常使用四次挥手：一方发送 FIN 包表示数据发送完毕，另一方回复 ACK 确认；然后另一方也发送 FIN 包，第一方回复 ACK 确认。双方都确认关闭后，连接完全断开。之后协议栈会删除套接字，释放占用的系统资源。
