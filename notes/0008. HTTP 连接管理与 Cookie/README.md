## 3. 用户与服务器的交互——Cookie

### 3.1. Cookie 的工作原理

HTTP 是无状态协议，但很多 Web 应用需要识别用户身份（如保持登录状态、记住购物车内容）。Cookie 技术正是为了解决这个问题而诞生的。

Cookie 的工作流程分为四步。当客户端首次访问服务器时，服务器在响应报文中添加一个 Set-Cookie 首部字段，其中包含一个唯一的标识符。客户端收到响应后，浏览器将这个 Cookie 保存在本地的 Cookie 文件中。之后客户端再次向该服务器发送请求时，浏览器会自动在请求报文中添加 Cookie 首部，把之前保存的标识符发送回去。服务器收到请求后，通过 Cookie 中的标识符识别出这是之前访问过的用户，从而实现状态跟踪。

### 3.2. Cookie 的作用

Cookie 在 Web 应用中有着广泛的用途。最常见的是用户认证——用户登录后，服务器通过 Cookie 标记会话状态，后续请求无需重复登录。购物车功能也依赖 Cookie，在用户浏览和添加商品的过程中保持购物车的内容。此外，Cookie 还用于个性化设置（如语言偏好、主题选择）、行为追踪（如广告定向）等场景。

### 3.3. Cookie 的安全与隐私

Cookie 本身是存储在客户端的纯文本信息，存在一定的安全和隐私风险。如果 Cookie 中包含敏感信息且未加密传输，可能被中间人截获。因此涉及敏感操作时应使用 HTTPS 加密连接。另外，第三方 Cookie（由用户访问的页面中嵌入的其他域设置的 Cookie）常被用于跨站追踪用户行为，引发了隐私方面的争议。现代浏览器已逐步限制第三方 Cookie 的使用。

## 4. HTTP 连接管理

### 4.1. 短连接与长连接

短连接是指每次 HTTP 请求都建立一个新的 TCP 连接，请求完成后立即关闭。HTTP/1.0 默认使用短连接。这种方式简单直接，但频繁的 TCP 三次握手和四次挥手带来了较大的性能开销。

长连接（也叫持久连接）是指在一个 TCP 连接上可以发送多个 HTTP 请求和响应。HTTP/1.1 默认使用长连接，除非显式声明 Connection: close。长连接减少了 TCP 连接建立和关闭的次数，降低了延迟和服务器负载。但长连接也有缺点——如果连接长时间空闲，会占用服务器资源，因此通常会设置超时时间来关闭不活跃的连接。

### 4.2. 流水线机制

流水线是 HTTP/1.1 引入的一种优化机制。在普通的长连接中，客户端必须等待收到前一个响应后才能发送下一个请求，这种方式称为非流水线。而流水线允许客户端在收到前一个响应之前就连续发送多个请求，服务器按顺序处理并返回响应。流水线减少了请求的等待时间，提高了连接的利用率。不过在实际应用中，由于队头阻塞问题（一个请求的延迟会阻塞后续所有请求），流水线的效果有限，这也是 HTTP/2 提出多路复用的原因之一。

### 4.3. 并行连接

并行连接是指浏览器同时打开多个 TCP 连接，每个连接独立发送请求。这是浏览器为了加快页面加载速度而采用的策略。当一个页面包含多个对象（如图片、样式表、脚本）时，浏览器不会串行地逐个下载，而是同时发起多个连接并行下载。现代浏览器通常会对同一个域名限制并行连接数（如 6 个），以避免给服务器造成过大压力。并行连接与长连接配合使用，可以在减少连接建立开销的同时充分利用网络带宽。
