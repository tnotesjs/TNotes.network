## 3. HTTP/2

### 3.1. 帧与消息

HTTP/2 最重要的变化之一是引入了帧（frame）的概念。在 HTTP/1.x 中，数据以文本格式的报文为单位传输。HTTP/2 将所有通信数据分割为更小的帧，每个帧都有固定的格式，包含长度、类型、标志位、流标识符和有效载荷。请求和响应消息被拆分成多个帧，这些帧可以在同一个连接上交错发送，在接收端再重新组装成完整的消息。这种二进制分帧的方式比 HTTP/1.x 的文本解析更高效，也更易于精确控制。

### 3.2. 多路复用

多路复用是 HTTP/2 解决队头阻塞问题的关键特性。在 HTTP/1.1 中，即使使用了长连接和流水线，同一个连接上的请求仍然必须按顺序处理，前面的请求延迟会阻塞后面的请求。HTTP/2 通过流（stream）的概念解决了这个问题。每个请求/响应对被分配一个唯一的流标识符，不同流的帧可以在同一个 TCP 连接上交错传输。客户端和服务器可以同时发送和接收多个流的帧，无需等待前面的请求完成。这使得一个 TCP 连接就能并行处理多个请求，大大提高了连接的利用率。

### 3.3. 服务器推送

服务器推送是 HTTP/2 引入的一项新能力。在传统的 HTTP 模式中，客户端请求一个 HTML 页面后，需要解析 HTML 才能发现其中引用的 CSS、JavaScript 等资源，然后逐个请求这些资源。HTTP/2 允许服务器在客户端请求 HTML 时，主动将相关的资源推送到客户端。例如，客户端请求 index.html 时，服务器可以同时推送 style.css 和 app.js，省去了客户端逐个发现和请求的往返时间。

### 3.4. 头部压缩

HTTP 请求和响应中包含大量的头部字段，这些头部在每次请求中往往高度重复（如 User-Agent、Cookie 等）。HTTP/2 使用 HPACK 压缩方案来减少头部的传输开销。HPACK 维护一个头部字段表，发送过的头部字段会被索引化，后续请求中相同的字段只需发送索引号即可。对于变化的字段值，使用霍夫曼编码进行压缩。这在移动端等带宽受限的场景下效果尤为明显。

### 3.5. HTTP/2 的局限性

HTTP/2 虽然解决了应用层的队头阻塞问题，但仍然依赖 TCP 作为传输层协议。TCP 本身存在队头阻塞——当一个 TCP 段丢失时，后续的所有数据都必须等待该段重传成功后才能交付给应用层。即使这些后续数据对应的可能是不同的 HTTP 流，也无法绕过 TCP 层的阻塞。此外，TCP 的三次握手和 TLS 的握手在建立连接阶段仍然需要额外的往返时间。

## 4. HTTP/3

### 4.1. 从 TCP 到 QUIC

为了从根本上解决 TCP 带来的队头阻塞问题，HTTP/3 选择了放弃 TCP，转而使用基于 UDP 的 QUIC 协议作为传输层。QUIC 在 UDP 之上实现了可靠传输、流量控制和拥塞控制等 TCP 的核心功能，同时避免了 TCP 的队头阻塞。QUIC 将传输层和 TLS 加密层合二为一，在建立连接时只需一次往返即可完成握手和加密协商。

### 4.2. QUIC 协议的特点

QUIC 有几个显著的特点。首先是多路复用且无队头阻塞——QUIC 中的流是独立的，一个流的数据丢失不会影响其他流的交付。其次是更快的连接建立——QUIC 将传输层握手和 TLS 握手合并，首次连接只需 1-RTT，后续连接可以实现 0-RTT。第三是连接迁移——QUIC 使用连接 ID 而非四元组来标识连接，当用户的网络环境切换时（如从 WiFi 切换到蜂窝网络），连接可以无缝迁移而不会断开。

### 4.3. HTTP/3 的优势

HTTP/3 在 HTTP/2 的基础上进一步提升了性能。在高丢包率的网络环境下，HTTP/3 的表现远优于 HTTP/2，因为单个数据包的丢失不会阻塞其他流。对于移动用户来说，连接迁移功能使得网络切换时不需要重新建立连接，用户体验更加流畅。0-RTT 的连接建立方式也显著降低了首次请求的延迟。不过 HTTP/3 也面临一些挑战，如 UDP 流量可能被某些防火墙限制、服务器端的实现复杂度更高等。
