## 3. UDP 的特点

### 3.1. 为什么需要 UDP

TCP 提供了可靠传输、流量控制和拥塞控制等完善的功能，但这些功能是有代价的——增加了延迟和协议开销。对于某些应用来说，这些代价是不可接受的。例如 DNS 查询只需要发送一个小请求并等待一个应答，使用 TCP 建立连接的开销（三次握手）远大于实际数据传输的开销。实时音视频应用宁愿容忍少量丢包也不希望因为重传导致延迟增加。因此，需要一种更轻量的传输协议，这就是 UDP 存在的意义。

### 3.2. UDP 的优点

UDP 有几个显著的优点。首先是开销小——UDP 首部只有 8 个字节，远小于 TCP 的 20 字节以上。其次是无连接——不需要三次握手和四次挥手，发送数据前无需建立连接，减少了延迟。第三是没有拥塞控制——UDP 可以以任意速率发送数据，这在某些实时应用中是一个优势。第四是支持一对多通信——UDP 支持广播和多播，而 TCP 只支持一对一通信。

### 3.3. UDP 的缺点

UDP 的缺点主要在于它不提供可靠性保障。UDP 不保证数据能够到达目的地、不保证数据按序到达、不保证不重复。如果发送端发送数据的速率过快，网络中的路由器会因为拥塞而丢弃分组，UDP 不会对此做任何处理。使用 UDP 的应用如果需要可靠性，必须在应用层自行实现重传、排序和流量控制等机制。

## 4. UDP 报文段结构

UDP 报文段的结构非常简洁，只有两个部分：首部和载荷。首部仅包含 4 个字段，每个字段占 2 个字节。源端口号标识发送方的应用进程，如果不需要返回数据可以设为 0。目的端口号标识接收方的应用进程。长度字段表示整个 UDP 报文段（包括首部和载荷）的字节数。检验和用于检测报文段在传输过程中是否出现了比特错误。

## 5. UDP 检验和

### 5.1. 检验和的计算方法

UDP 检验和的计算方法是将报文段的内容视为 16 比特整数的序列，对所有整数求和。如果总和超过 16 位，将溢出的高位回卷（wrap around）并与低位相加。最后对结果取反码，得到检验和。发送方将计算出的检验和填入首部的检验和字段。接收方收到报文段后，用同样的方法对整个报文段（包括检验和字段）求和，如果结果全为 1 则说明没有检测到错误。

### 5.2. 检验和的作用

UDP 检验和提供了一种端到端的差错检测机制。它的目的是在数据从发送端到接收端的整个传输路径上检测是否发生了比特翻转。虽然链路层也有自己的差错检测（如 CRC），但那只能保证单条链路上的正确性。经过路由器转发时，数据会从一个链路层帧中取出再放入另一个帧，链路层的检验和会被重新计算。UDP 的检验和确保了端到端的完整性。
