## 3. 可靠数据传输原理

### 3.1. 构造可靠数据传输协议

可靠数据传输协议（rdt）是计算机网络中最基本、最重要的协议之一。它的目标是在不可靠的底层信道上提供可靠的数据传输服务。构造 rdt 协议的过程遵循增量式设计方法——从最简单的情况开始，逐步增加复杂性来应对更恶劣的信道条件。

### 3.2. 经完全可靠信道的可靠数据传输——rdt1.0

在最简单的情况下，底层信道是完全可靠的，不会出现比特错误和丢包。此时 rdt1.0 非常简单：发送方从上层接收数据，封装成分组发送到信道上；接收方从信道接收分组，提取数据交给上层。不需要任何确认、重传或差错检测机制。

### 3.3. 经具有比特差错信道的可靠数据传输——rdt2.0

当底层信道可能出现比特错误时，协议需要增加差错检测和纠正机制。rdt2.0 引入了三个新机制：检验和用于检测比特错误；肯定确认（ACK）由接收方发送，告诉发送方数据已正确收到；否定确认（NAK）告诉发送方数据有误，需要重传。这种基于重传的机制称为 ARQ（自动重传请求）协议。但 rdt2.0 有一个致命缺陷——ACK 或 NAK 本身也可能出错，rdt2.1 和 rdt2.2 通过序号和肯定确认解决了这个问题。

### 3.4. 经具有比特差错的丢包信道的可靠数据传输——rdt3.0

当信道不仅可能出现比特错误，还可能丢失分组时，协议还需要处理丢包问题。rdt3.0 引入了倒计数定时器——发送方在发送分组时启动定时器，如果在超时时间内没有收到 ACK，就认为分组丢失并重传。由于发送方无法区分是分组丢失了还是 ACK 丢失了，它总是重传最近发送的分组，接收方通过序号来判断是否是重复分组并丢弃重复。

## 4. 流水线可靠数据传输协议

### 4.1. 流水线的基本概念

rdt3.0 的性能很差——发送方每发送一个分组就必须等待 ACK，信道利用率极低。流水线技术允许发送方在等待 ACK 之前连续发送多个分组，充分利用信道带宽。要实现流水线，协议需要增加序号的范围（因为同时有多个未确认的分组）、在发送方和接收方增加缓存。

### 4.2. 回退 N 步

回退 N 步（GBN）协议是流水线协议的一种实现。它允许发送方连续发送最多 $N$ 个未确认的分组，$N$ 称为窗口大小。发送方维护一个发送窗口，只允许发送序号落在窗口内的分组。接收方采用累积确认——收到序号为 $n$ 的分组后，确认号表示 $n$ 之前（含 $n$）的所有分组都已正确收到。如果某个分组丢失，接收方会丢弃该分组之后的所有失序分组，发送方超时后从丢失的那个分组开始重传窗口内的所有分组。

### 4.3. 选择重传

选择重传（SR）协议是另一种流水线实现，它只重传丢失的分组而非整个窗口。发送方为每个分组维护独立的定时器，哪个分组超时就只重传哪个。接收方为每个分组维护一个接收窗口，能够缓存失序到达的分组。当缺失的分组到达后，接收方将缓存中连续的分组一起交付给上层。SR 的窗口大小必须满足 $N \leq 2^k$（$k$ 为序号位数），否则会出现序号歧义。

## 5. TCP 的可靠数据传输

### 5.1. 超时间隔加倍

TCP 使用与 rdt3.0 类似的机制来实现可靠传输，但也有一些重要的改进。当发生超时时，TCP 不仅重传最早的未确认报文段，还会将超时间隔加倍。这种指数退避策略的目的是在网络拥塞时减缓重传速率，避免加重拥塞。每当收到新的 ACK（确认了新数据）时，超时间隔会重新计算恢复到正常值。

### 5.2. 快速重传

如果仅依赖超时来检测丢包，等待时间可能过长。TCP 引入了快速重传机制——当发送方连续收到 3 个冗余 ACK（即同一个确认号的第 4 次出现）时，就认为该确认号对应的报文段已经丢失，立即重传该报文段，而不必等待超时。冗余 ACK 表明接收方收到了失序的报文段，间接暗示某个中间报文段可能丢失了。

### 5.3. 是回退 N 步还是选择重传

TCP 的可靠传输机制介于 GBN 和 SR 之间。它像 GBN 一样使用累积确认，接收方只确认按序到达的最高序号。但它又像 SR 一样能够缓存失序到达的报文段，避免不必要的重传。当收到失序报文段时，TCP 会立即发送一个冗余 ACK，但不会丢弃失序数据。这种设计在实现复杂度和性能之间取得了良好的平衡。
