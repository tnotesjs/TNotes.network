## 3. 流量控制

### 3.1. 为什么需要流量控制

发送方和接收方的数据处理速度往往不同。如果发送方发送数据的速度超过了接收方读取和处理数据的速度，接收方的接收缓冲区就会被填满，后续到达的数据将被丢弃。流量控制机制的目的是让发送方根据接收方的实际接收能力来调节发送速率，防止接收方溢出。

### 3.2. 接收窗口

TCP 通过接收窗口（rwnd）来实现流量控制。接收方在每个 ACK 报文段中通告自己的接收窗口大小，告诉发送方「我的接收缓冲区还有多少空间」。接收窗口的值等于接收缓冲区的总大小减去已占用的空间。发送方必须保证已发送但未确认的数据量不超过接收方通告的接收窗口大小。

### 3.3. 滑动窗口机制

流量控制通过滑动窗口机制来实现。发送方维护一个发送窗口，窗口的左边界是最早未确认的字节序号，右边界是最早未确认的字节序号加上接收窗口的大小。发送方只能发送序号落在窗口内的数据。当收到新的 ACK 后，窗口的左边界向右滑动；当收到新的接收窗口通告后，窗口的右边界可能向右滑动（窗口增大）或保持不变（窗口不变）。

## 4. 滑动窗口详解

### 4.1. 发送窗口

发送窗口将字节流分为四个区域：已发送且已确认的字节（窗口左侧）、已发送但未确认的字节（窗口内前半部分）、允许发送但尚未发送的字节（窗口内后半部分）、不允许发送的字节（窗口右侧）。发送方在发送新数据时，从未确认的最小序号之后开始填充窗口。收到 ACK 后，窗口左边界向右滑动，释放出空间允许发送更多数据。

### 4.2. 接收窗口

接收窗口的左边界是期望收到的下一个按序字节的序号。接收方会缓存失序到达的报文段（它们落在接收窗口内但在左边界右边），但只有当左边界处的字节到达后，才会将连续的数据交付给应用层并滑动窗口。接收窗口的大小随应用层读取数据的速度而变化——应用层读取得快，窗口就能更快地释放空间。

### 4.3. 窗口滑动过程

窗口滑动是 TCP 流量控制的核心动作。在发送端，每当收到一个确认新数据的 ACK，发送窗口就向右滑动，被确认的字节移出窗口，新的字节进入窗口。在接收端，每当将数据交付给应用层，接收窗口就向右滑动。两端的窗口大小可以动态变化，当接收方的应用层暂时停止读取数据时，接收窗口会逐渐缩小到 0，发送方将停止发送。

### 4.4. 糊涂窗口综合征

糊涂窗口综合征（Silly Window Syndrome）是流量控制中可能出现的一个性能问题。当接收方每次只读取很少量的数据时，会频繁地通告很小的接收窗口；发送方收到小窗口通告后，会发送很小的报文段。大量小报文段导致网络效率极低（首部开销占比过高）。解决方案有两方面：接收方延迟发送小窗口通告，直到窗口增长到一定大小；发送方使用 Nagle 算法，等到数据量足够大或收到 ACK 后再发送。
