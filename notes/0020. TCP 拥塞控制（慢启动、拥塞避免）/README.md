## 3. 拥塞原因与代价

### 3.1. 两个发送方和一台具有无穷大缓存的路由器

在最简单的场景中，两个发送方共享一条链路，路由器有无穷大的缓存。随着发送速率增加，链路达到饱和后，分组开始在路由器中排队。虽然不会丢包，但排队时延会随着发送速率接近链路容量而趋向无穷大。这个代价看似温和，但实际上意味着网络性能严重退化。

### 3.2. 两个发送方和一台具有有限缓存的路由器

当路由器的缓存有限时，拥塞的代价更加严重。当到达速率超过输出链路容量时，缓存会被填满，后续的分组将被丢弃。发送方超时后重传这些分组，但问题在于发送方无法区分「分组在路由器被丢弃」和「ACK 在返回途中丢失」，可能对同一个分组发送多份副本，浪费了宝贵的链路带宽。拥塞越严重，重传越多，浪费越大。

### 3.3. 四个发送方和具有有限缓存的多台路由器

在更现实的多跳场景中，拥塞的代价更加复杂。一个分组可能经过多台路由器，只要其中任何一台发生拥塞，分组就可能被丢弃。当某个发送方的分组在上游路由器被丢弃时，它在到达该路由器之前经过的所有链路的传输能力都被浪费了。多跳路径上的竞争使得拥塞的代价成倍放大。

## 4. 拥塞控制方法

### 4.1. 端到端拥塞控制

端到端拥塞控制不需要网络层提供显式的拥塞信号，发送方通过观察网络行为来推断是否发生了拥塞。TCP 使用这种方法——通过观察丢包事件（超时或收到 3 个冗余 ACK）来判断网络可能拥塞了，从而降低发送速率。端到端方法的优点是不需要修改路由器，部署简单。

### 4.2. 网络辅助的拥塞控制

网络辅助的拥塞控制中，路由器直接向发送方提供拥塞反馈信号。路由器可以在转发的分组中设置一个拥塞指示比特，或者直接向发送方发送一个特殊的拥塞分组。显式拥塞通告（ECN）就是网络辅助拥塞控制的一个例子——路由器在检测到拥塞时在 IP 首部设置 ECN 标志，接收方在 ACK 中告知发送方。

## 5. TCP 拥塞控制

### 5.1. 经典的 TCP 拥塞控制

TCP 的拥塞控制通过维护一个拥塞窗口（cwnd）来实现。发送方的发送速率受 cwnd 和 rwnd 中较小者的限制：已发送但未确认的数据量不超过 min(cwnd, rwnd)。TCP 通过调节 cwnd 的大小来控制注入网络的数据量。经典算法包含三个阶段：慢启动、拥塞避免和快速恢复。

### 5.2. 慢启动

TCP 连接刚建立时，cwnd 通常设置为 1 个 MSS（最大报文段大小）。在慢启动阶段，每收到一个 ACK，cwnd 就增加 1 个 MSS。这意味着每个 RTT 内 cwnd 会翻倍——指数增长。慢启动并不「慢」，它会让 cwnd 快速增长。当 cwnd 达到慢启动阈值（ssthresh）时，TCP 进入拥塞避免阶段。如果检测到丢包（3 个冗余 ACK），ssthresh 设为 cwnd 的一半，cwnd 减半并进入快速恢复；如果超时，ssthresh 设为 cwnd 的一半，cwnd 重置为 1 并重新进入慢启动。

### 5.3. 拥塞避免

一旦 cwnd 达到 ssthresh，TCP 进入拥塞避免阶段。与慢启动的指数增长不同，拥塞避免阶段采用线性增长——每个 RTT 内 cwnd 增加 1 个 MSS（实现上通常是每个 ACK 增加 MSS/cwnd）。这种更保守的增长策略是为了避免在接近网络容量时加剧拥塞。如果在拥塞避免阶段检测到丢包，cwnd 和 ssthresh 会相应调整。

### 5.4. 快速恢复

快速恢复是在收到 3 个冗余 ACK 时触发的拥塞响应机制。与超时不同，3 个冗余 ACK 暗示网络还有一定的传输能力（否则不会收到后续的 ACK）。在快速恢复中，ssthresh 设为 cwnd 的一半，cwnd 设为 ssthresh 加上 3 个 MSS（因为收到了 3 个冗余 ACK 说明有 3 个分组离开了网络），然后进入拥塞避免阶段。这比超时后从 cwnd = 1 重新开始要温和得多。

### 5.5. TCP 拥塞控制的 FSM 描述

TCP 拥塞控制的状态机可以描述为三种状态之间的转换。慢启动状态中 cwnd 指数增长，达到 ssthresh 时转入拥塞避免。拥塞避免状态中 cwnd 线性增长。当检测到超时时，cwnd 回到 1 个 MSS 并进入慢启动。当收到 3 个冗余 ACK 时，进入快速恢复。从快速恢复出来后进入拥塞避免。这个状态机在 TCP Reno 和 TCP Tahoe 等版本中有不同的实现细节。
